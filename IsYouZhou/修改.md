# 修改说明

## 本次目标
- 在现有大地图系统上增加第二种地图模式。
- 新模式下玩家通过 `W/A/S/D` 自由移动。
- 入口点 `enter` 不再默认按原模式直接激活显示，而是改为玩家靠近节点时显示。
- 不修改原有 `AllMapBase` 的核心逻辑，不改地图资源结构，只通过单独模式文件决定哪些场景启用新模式。

## 实际修改文件

### 1. `Scene/walkMap/WalkMapModeConfig.cs`
- 作为 WalkMap 模式的唯一场景判断文件。
- 通过 `EnabledSceneNames` 控制哪些场景启用新模式。
- 未加入列表的场景，继续走原地图模式。

### 2. `Scene/walkMap/WalkMapController.cs`
- 新增 WalkMap 模式控制器。
- 实现内容：
  - 读取 `W/A/S/D` 控制玩家移动。
  - 禁用原有 `AllMapClick`，避免鼠标点击节点移动和新模式冲突。
  - 入口点仍按原结构查找：`transform.Find("enter")`。
  - 根据玩家和节点的距离控制 `enter` 显示与隐藏。
  - 玩家靠近节点时，同步该节点到 `NowIndex`，保证入口交互仍然能按原节点逻辑工作。

## 本次新增的边界与视野修正
- 以 `AllMap/MapLand` 的实际范围作为移动和相机的基础边界。
- 玩家可移动区域在 `MapLand` 范围内再减去一圈内边距，避免贴边移动。
- 主相机正交大小基于 `MapLand` 实际大小重新计算，并额外扣除可见边距，避免初始视野过大。
- 主相机位置会持续限制在 `MapLand` 的安全范围内，减少页面抖动问题。

## 3. `Scene/YZSceneManeger.cs`
- 保留原有场景加载流程。
- 抽出公共创建方法，在创建 `AllMapBase` 后判断：
  - 如果当前场景命中 `WalkMapModeConfig`，则额外挂载 `WalkMapController`
  - 否则保持原模式
- 这是新模式唯一的接入点。

## 4. `YouZhou.csproj`
- 将以下文件加入编译：
  - `Scene/walkMap/WalkMapController.cs`
  - `Scene/walkMap/WalkMapModeConfig.cs`

## 未修改内容
- `Scene/AllMapBase.cs` 未改。
- 原地图节点、`enter` 子物体、地图资源本身未改。
- 原地图模式的行为仍保留。

## 使用方式
1. 打开 `Scene/walkMap/WalkMapModeConfig.cs`
2. 在 `EnabledSceneNames` 中加入需要启用 WalkMap 的场景名
3. 进入该场景后自动启用第二种地图模式

## 当前结果
- 新模式已经具备独立接入能力。
- 原模式没有被直接改写。
- WalkMap 模式下的移动、入口显隐、节点同步、相机范围和移动边界都已经接入完成。

# 2026/04/20 - AllMapComponent 寻路逻辑整合

## 目标
将参考版（`参考/AllMapComponent.cs`）的原版寻路逻辑抄写到自己的 `Scene/AllMapComponent.cs` 中。
参考版核心特点：通过 `MapGetWay` 计算路径、逐节点分段移动、支持中断检测。

## 修改文件

### `Scene/AllMapComponent.cs`

#### 1. 新增引用
```csharp
using GetWay;
```

#### 2. `AvatarMoveToThis()` 逻辑变更
**修改前：**
- 没有遁术 + 可移动 → 调用 `MoveToNode(this.NodeIndex, false)`（自己写的 BFS + `MoveAlongPath` 一次性走完）

**修改后：**
- 没有遁术 + 可移动 → 调用 `base.StartCoroutine(this.Move())`（原版逐节点协程寻路）
- 有遁术 → 保持 `MoveToNode(this.NodeIndex, true)` 直接飞行

#### 3. 新增 `NewEventRandom()`（参考原版）
```csharp
public new virtual void NewEventRandom()
{
    bool flag = !base.CanClick();
    if (!flag)
    {
        this.fuBenSetClick();
        base.NewMovaAvatar();
    }
}
```
作用：检查可点击状态后，设置副本点击状态并执行单段移动。

#### 4. 新增 `Move()` 协程（核心寻路逻辑）
```csharp
public virtual System.Collections.IEnumerator Move()
```
逻辑流程：
1. 通过 `MapGetWay.Inst.IsNearly()` 判断起点和目标是否相邻
2. 相邻则路径只有目标节点；否则调用 `MapGetWay.Inst.GetBestList()` 计算最佳路径
3. 路径长度大于 1 时显示移动提示（`MapMoveTips.Show()` 已注释，项目中无此类）
4. 遍历路径中的每个节点：
   - 静态节点 → 调用该节点的 `NewMovaAvatar()`
   - 非静态节点 → 调用该节点的 `NewEventRandom()`
5. 每段移动后等待 `AllMapManage.instance.isPlayMove` 结束
6. 检测中断条件：
   - `!Tools.instance.canClick(false, true)`（玩家不可操作）
   - `MapGetWay.Inst.IsStop`（外部停止标记）
7. 结束时调用 `MapGetWay.Inst.StopAuToMove()` 清理状态

#### 5. `EventRandom()` 逻辑变更
**修改前：**
```csharp
this.AvatarMoveToThis();
this.BaseAddTime();
```

**修改后：**
- 遁术状态 → 调用 `this.NewEventRandom()`
- 普通状态 → 调用 `base.StartCoroutine(this.Move())`

## 保留的原有方法
以下方法仍然保留，继续用于遁术移动和命令调用：
- `MoveToNode(int targetNodeIndex, bool useDunShu)`
- `MoveAlongPath(List<int> path, MapPlayerController playerController, BaseMapCompont startComp)`
- `FindPath(int startIndex, int targetIndex)`（自己写的 BFS）
- `CommandMoveToNode(int targetNodeIndex)`
- `SetAvatarNowMapIndexStatic(int nodeIndex)`

## 可能存在的编译问题
| 符号 | 来源 | 说明 |
|---|---|---|
| `this.IsStatic` | `MapComponent` 基类 | 参考版使用了该属性，若基类没有会报错 |
| `this.fuBenSetClick()` | `MapComponent` 基类 | 原版调用，若基类没有会报错 |
| `base.NewMovaAvatar()` | `MapComponent` 基类 | 原版调用，若基类没有会报错 |
| `MapMoveTips.Show()` | 原版 UI 类 | 项目中无此类，已在代码中注释掉 |
| `MapGetWay.Inst.StopAuToMove()` | `MapGetWay` 类 | 需确认 `MapGetWay` 有此实例方法 |

## 核心差异总结
| | 原有逻辑 | 整合后逻辑 |
|---|---|---|
| 寻路算法 | 自己写 `FindPath()` BFS 遍历 `IsToolsMain.MapMoveDatas` | 调用原版 `MapGetWay.Inst.GetBestList()` |
| 移动方式 | `MoveAlongPath()` 一次性走完所有节点 | `Move()` 协程逐节点分段移动 |
| 中断检测 | 无 | 每段移动后检测 `canClick` 和 `IsStop` |
| 遁术处理 | 在 `AvatarMoveToThis()` 中判断 | 分离到 `NewEventRandom()` 和 `Move()` 中分别处理 |
